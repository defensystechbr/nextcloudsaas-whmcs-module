# Análise de impacto: Manager v12.x → Módulo WHMCS v3.2.0

> Documento de planejamento. Compara o estado atual do `nextcloud-saas-manager`
> (HEAD `10bfd99`, releases v12.0/v12.1/v12.2 + correções v11.3.x) contra o
> módulo WHMCS atual (`v3.1.7`) e descreve o roadmap proposto para a próxima
> versão do módulo (`v3.2.0`).

---

## 1. Resumo executivo

O `manage.sh` deu um salto qualitativo entre v11.3.4 e v12.2: ganhou modo
**assíncrono completo**, **gateway SSH endurecido**, **idempotency keys**,
**webhooks de conclusão**, **JSON estruturado em quase todos os subcomandos**,
**namespaces hierárquicos** (`user`, `group`, `apps`, `occ-exec`),
**health check consolidado**, e **backup off-site via restic** para S3/B2 com
agendamento por systemd timer.

O módulo WHMCS, por outro lado, ainda está no contrato v11.x: conecta como
`root`, executa o `manage.sh` em modo síncrono via `bash <script> <client>
<domain> <cmd>`, parseia output texto, e implementa internamente coisas que
agora o servidor já oferece nativamente (lookup de Custom Field, fallback de
HaRP key, `instanceExists`).

A conclusão é que **vale a pena uma versão maior** — `v3.2.0` — que migra o
módulo para o contrato JSON-first do manager v12.x. Em vez de quebrar a
compatibilidade do dia para a noite, propomos uma **migração progressiva em
camadas** dentro de uma única release:

1. Manter o atual `runManage()` como caminho legado (fallback) para servidores
   ainda na v11.x.
2. Detectar a versão do servidor (`nextcloud-manage health --json`) na conexão
   inicial e armazenar em cache no `tblservers`.
3. Quando `>= 12.0`, usar **JSON estruturado**, **modo `--async`**,
   **idempotency keys** e **`ncsaas-api` shim** para tudo.
4. Quando `< 12.0`, manter o caminho atual (texto + síncrono + root).

Esta abordagem evita um "big bang" e permite que provedores que ainda não
atualizaram o servidor não sejam quebrados pela atualização do módulo.

## 3. Gaps identificados no módulo WHMCS atual (v3.1.7)

### 3.1 Conexão SSH

- **Conecta como `root`** com password ou chave; não usa o usuário dedicado
  `ncsaas-api`. Risco de segurança e impossível de auditar.
- O shim `ncsaas-api-shim` rejeita metacaracteres de shell. Algumas chamadas
  do módulo usam `bash -c '...'` com pipes/redirects (`grep | head`,
  `docker exec ... env | grep`) que serão **bloqueadas** se trocadas para
  `ncsaas-api`. Precisamos refatorar para usar verbos da allowlist.
- A leitura direta de arquivos via `cat /opt/.../.credentials` deixa de
  funcionar com o shim — precisamos usar `nextcloud-manage <client> _ credentials`.

### 3.2 Provisionamento síncrono

- `nextcloudsaas_CreateAccount` chama `runManage('create')` síncrono via SSH.
  Como `create` leva 5–15 min, o WHMCS frequentemente atinge timeout do
  módulo (PHP `max_execution_time` é tipicamente 30s em hosts compartilhados).
- O `AfterCronJob` mascara o problema fazendo polling, mas o usuário admin
  que clica em **Module → Create** vê erro mesmo quando o create vai funcionar.

### 3.3 Falta de uso de JSON

- `runManage` interpreta `success` apenas pelo exit code do shell, não pelos
  códigos do contrato (`client_already_exists` = 1, `idempotency_conflict` = 3,
  validação = 5, `not_implemented_yet` = 99).
- O parsing de `.credentials` é via regex em texto livre. Quando o `manage.sh`
  v11.x mudou o formato, quebrou silenciosamente (foi o que motivou v3.1.6).
  Migrar para `nextcloud-manage <client> _ credentials --json` (quando
  disponível) elimina essa fragilidade.

### 3.4 Falta de idempotência

- Se o cron `AfterCronJob` rodar duas vezes em paralelo (raro, mas possível
  com bug no WHMCS), o módulo dispara dois `manage.sh create` para o mesmo
  tenant. A v3.1.5 mitiga via `instanceExists`, mas o servidor agora oferece
  garantia forte via `--idempotency-key`.

### 3.5 Polling em vez de webhooks

- O `AfterCronJob` roda a cada 5 minutos perguntando "está pronto?". Com
  webhook, o servidor avisa o WHMCS no instante da conclusão do job
  (latência típica < 1 s). Reduz carga no servidor e dá UX melhor ao admin.

### 3.6 Funcionalidades não expostas

| Funcionalidade do servidor | Exposição no módulo hoje |
|---|---|
| `backup-offsite` (S3/B2) | Não exposta. Admin precisa de SSH manual. |
| `health` consolidado | Não exposta. Admin Test Connection só checa SSH. |
| `worker status` | Não exposta. |
| `job list/logs/cancel` | Não exposta. |
| `upgrade-harp` | Não exposta. |
| `user/group/apps namespaces` | Não exposta. Hooks `ChangePackage` não fazem nada. |
| `occ-exec` (reset password etc.) | Parcialmente, via método ad-hoc. |

## 4. Plano da release v3.2.0

### 4.1 Princípios

1. **Backward-compatible:** servidores v11.x continuam funcionando.
2. **JSON-first:** sempre que o servidor for >= v12.0, falar JSON.
3. **Async-first:** `create`/`backup`/`restore`/`remove`/`update` enfileiram
   job; o módulo armazena `job_id` em Service Custom Field; webhook ou cron
   atualiza o `domainstatus` quando concluir.
4. **Defesa em profundidade:** mesmo no fluxo async, o `AfterCronJob` continua
   como rede de segurança (caso o webhook falhe).

### 4.2 Lista de mudanças propostas

#### A. Detecção de versão e capabilities

- Novo método `SSHManager::detectServerCapabilities()` que executa
  `nextcloud-manage health --json` (com fallback para `bash manage.sh
  --version` quando o JSON não retornar) e retorna `array {
  manager_version: "12.2.0", supports_async: true, supports_json: true,
  supports_backup_offsite: true, supports_health_json: true,
  shim_user: "ncsaas-api"|"root" }`.
- Cache em memória por requisição (e opcionalmente em `tblservers.notes`
  para evitar 1 SSH extra por chamada).

#### B. Conexão dual: `root` + `ncsaas-api`

- Novo campo no Server Profile: **"SSH user para API"** (default `root`,
  recomendado `ncsaas-api`).
- Novo botão admin **"Verificar Gateway"**: confirma que `ncsaas-api` está
  configurado e que a chave do WHMCS está em `/home/ncsaas-api/.ssh/authorized_keys`.
- Quando `shim_user = ncsaas-api`, o `executeCommand` deixa de usar `bash -c`
  e passa apenas `nextcloud-manage <args>`.

#### C. Modo assíncrono + tracking de jobs

- Novo Custom Field interno do módulo (auto-criado): `nextcloud_job_id`
  (pending job em andamento) e `nextcloud_last_job_state` (queued/running/done/failed).
- `nextcloudsaas_CreateAccount` (quando server >= 12.0):
  1. Gera UUID v4 lowercase para `--idempotency-key`.
  2. Executa `nextcloud-manage <client> <domain> create --async --json
     --idempotency-key=<uuid> --callback=https://<whmcs>/modules/servers/nextcloudsaas/webhook.php?token=<hmac>`.
  3. Recebe `{ job_id }`, grava em Custom Field, retorna `success` imediatamente
     ao WHMCS (Order já vira Active, sem esperar).
- Endpoint webhook em `modules/servers/nextcloudsaas/webhook.php` valida
  HMAC, atualiza estado e dispara hook `JobCompleted` no WHMCS.
- `AfterCronJob` continua rodando como rede de segurança: a cada 5 min,
  para cada serviço com `nextcloud_job_id` definido e estado != done/failed,
  consulta `nextcloud-manage job <id> status --json` e atualiza.

#### D. JSON-first em parsing

- Novo método `SSHManager::getCredentialsJson($clientName)` tenta
  `nextcloud-manage <client> _ credentials --json` primeiro; se exit != 0,
  cai para o parser texto v3.1.7 atual.
- Idem para `statusInstance`, `verifySharedServices` (via `health --json`).

#### E. Novos botões admin

| Botão | Ação |
|---|---|
| **Health Check** | Executa `nextcloud-manage health --json` e renderiza painel com 8 checks. |
| **Backup Off-site** | Executa `nextcloud-manage <client> _ backup-offsite --json`. |
| **Habilitar Backup Diário** | Habilita timer systemd `nextcloud-saas-backup@<cliente>.timer` (via comando wrapper). |
| **Atualizar Apps** | `nextcloud-manage <client> apps enable groupfolders --async --json`. |
| **Reset Password** | `nextcloud-manage <client> occ-exec user:resetpassword <user> --payload-stdin --json`. |
| **Upgrade HaRP** | `nextcloud-manage <client> _ upgrade-harp --dry-run --json` + opção de aplicar. |
| **Ver Jobs** | Lista os últimos 20 jobs do tenant via `job list --client=<client> --json`. |

#### F. Hooks de mudança de pacote (`ChangePackage`)

- `nextcloudsaas_ChangePackage` passa a chamar `apps enable/disable` quando o
  novo pacote tem features diferentes (ex: pacote "Pro" habilita `groupfolders`).

#### G. Idempotency em todos os state-changing

- `create`, `remove`, `backup`, `restore`, `start`, `stop`, `update`,
  `backup-offsite` passam a sempre incluir `--idempotency-key=<uuid>`.
  UUID gerado e gravado em Service Custom Field para suportar retry.

#### H. Remover código que ficou redundante

- `SSHManager::getHarpSharedKey` continua para fallback v11.x mas deixa de
  ser necessário em v12.x (`credentials --json` virá com `harp_shared_key`).
- Parser regex de `.credentials` vira fallback "modo v11".

### 4.3 Compatibilidade e migração

| Cenário | Comportamento v3.2.0 |
|---|---|
| Servidor v11.x + módulo v3.2.0 | Modo síncrono + texto + root (legado v3.1.7). Funciona igual. |
| Servidor v12.x + módulo v3.2.0 (config default) | Detecta v12, pede ao admin para configurar `ncsaas-api` no Server Profile. Até configurar, continua usando `root` + JSON quando possível. |
| Servidor v12.x + `ncsaas-api` configurado | Modo `--async` + `--json` + webhook + idempotency. |

### 4.4 Itens fora do escopo da v3.2.0 (deferidos para v3.3.0)

- Multi-server load balancing (escolha do servidor menos carregado no checkout).
- Painel cliente listando seus próprios jobs (UI).
- Suporte a `--strict` em `apps enable` para rollback automático.
- i18n PT-PT / EN.
